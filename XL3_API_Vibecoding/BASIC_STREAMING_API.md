# XL3 Basic Streaming API

## Purpose

This document defines the XL3 Basic Streaming API for read-only live measurement streaming.

## Context

When the XL3 is in Sound Level Meter mode, it can push live measurement results in JSON format to up to 20 connected clients.

---

## Scope

- **Target:** XL3 Basic Streaming API live endpoint
- **Direction:** Client connects to the XL3 stream source and receives recurring measurement data
- **Access mode:** Read-only
- **Supported data families:**
  - `123` settings
  - `123` data
  - `spectrum` settings
  - `spectrum` data

## Firmware Compatibility

This document applies to XL3 firmware version 1.48 or higher.

---

## Endpoint

**XL3 Connect:**
```
wss://connect.nti-audio.com/<ConnectKey>/api/live/
```

**LAN:**
```
ws://<ip-or-host>/api/live/
```

---

## Preconditions

- The XL3 webserver must be running.
- The client authenticates with a password that is configured on the XL3 by the user.
- The API is read-only and cannot change XL3 settings.
- Successful Basic Streaming requires the XL3 to be in Sound Level Meter mode.

---

## Startup Sequence

After opening the WebSocket connection, the client shall use the following sequence:

1. Wait for the text prompt `Password:`.
2. Send the password followed by LF: `<Password>\n`
3. Send exactly one supported `START` command followed by LF.
4. Expect the first settings message as the first application JSON message.

No separate success acknowledgment is sent between command submission and the first settings message.

**Supported `START` commands:**

| Command | Description |
|---|---|
| `START 123 Spectrum` | 123 data and spectrum data |
| `START 123` | 123 data only |

---

## Message Model

All application messages are JSON objects using one of two top-level shapes.

**Settings message:**
```json
{
  "error": "NONE",
  "settings": {}
}
```

**Data message:**
```json
{
  "data": {}
}
```

---

## Message Semantics

### Settings message

A settings message defines the current stream structure and the metadata required to interpret and display subsequent data messages. It is sent:

- once at the beginning of a successful connection
- again whenever the relevant XL3 settings are changed on the device

The latest received settings message is the authoritative description of the active stream layout. The client shall derive its display structure from it, including where present: measurement names, measurement order, limit information, timer context, and spectrum metadata (including `octres`).

The client shall keep the current display structure, labels, and formatting persistent until a newer settings message is received.

### Data message

A data message contains the current live values belonging to the latest active settings message. The client shall interpret and display data according to the latest settings message and shall not hardcode measurement names, order, limits, timer configuration, or spectrum configuration.

---

## Update Rate

After successful startup, data messages are sent nominally every 500 ms. Settings messages may also be received during the connection whenever the user changes relevant settings on the XL3.

---

## Mapping Rules

### `123` mapping

The order of values in `data["123"]` corresponds to the order of descriptors in `settings["123"]`. For each index `i`:

- `settings["123"][i]` describes `data["123"][i]`

### `spectrum` mapping

If spectrum streaming is enabled, `settings.spectrum` describes the spectrum payload.

| Field | Currently defined values |
|---|---|
| `octres` | `1/1` or `1/3` |
| `bands` | `12` or `36` |

The spectrum array is expected to contain `settings.spectrum.bands` values. The client shall follow the `octres` and `bands` values from the latest settings message and must handle array length mismatches defensively without crashing.

---

## Formatting Rules

### Name formatting

Before display, names shall be transformed as follows:

```js
name.replace(/`/g, "<sub>").replace(/´/g, "</sub>")
```

### HTML safety

- All other HTML special characters must be escaped.
- Only `<sub>` tags generated by the transformation above are permitted.

### Null and value safety

- `null` values are valid stream conditions.
- Clients must never throw due to `null`, `undefined`, or non-numeric values.
- Display a placeholder such as `--` when a value is not displayable.
- Raw state handling and display formatting should be kept separate.

---

## Message Content Reference

### Settings message

A settings message can describe:

- device name
- running state
- timer configuration
- ordered `123` measurement descriptors with limit information
- spectrum metadata including `octres`

Example:
```json
{
  "error": "NONE",
  "settings": {
    "device_name": "Public XL3",
    "running": false,
    "timer": {"mode": "REPEATED_SYNC", "set": "00:30:00"},
    "123": [
      {"name": "L`Aeq´ +k1",    "limit": {"on": true,  "red": 99.0,  "amber": 95.0}},
      {"name": "L`Aeq´ +k1",    "limit": {"on": true,  "red": 99.0,  "amber": 95.0}},
      {"name": "L`CPKmax´ +k2", "limit": {"on": true,  "red": 135.0, "amber": 131.0}},
      {"name": "L`AFmin",       "limit": {"on": false, "red": 100.0, "amber": 90.0}},
      {"name": "L`AFmax",       "limit": {"on": false, "red": 100.0, "amber": 90.0}}
    ],
    "spectrum": {"name": "live", "octres": "1/3", "bands": 36}
  }
}
```

### Data message

A data message can contain:

- current timer value
- current measurement time
- ordered `123` values
- ordered spectrum values

Example:
```json
{
  "data": {
    "timer": "00:03:17",
    "measurement_time": "00:00:00",
    "123": [80.9, 80.9, 99.0, 26.0, 85.0],
    "spectrum": [

      74.0, 71.8, 71.5, 69.4, 72.9, 71.3, 74.8, 72.2, 73.4, 72.3,
      73.1, 72.0, 72.3, 72.5, 72.8, 73.0, 72.8, 72.9, 72.9, 72.3,
      73.1, 72.5, 72.7, 72.4, 72.7, 72.6
    ]
  }
}
```

## Startup and Error Conditions

### Startup text responses

Before application JSON messages begin, the server may send plain text responses:

| Response | Meaning |
|---|---|
| `Password:` | Credential prompt. The client shall wait for this before sending the password. |
| `Incorrect password` | The provided password was rejected. No application JSON stream is started. |
| `Already in use` | The maximum number of concurrent clients is already reached. No application JSON stream is started. |

If `Incorrect password` or `Already in use` is received, the client shall treat the connection attempt as unsuccessful.

### JSON-level status field

Settings messages use the top-level `error` field to communicate stream status:

| Value | Meaning |
|---|---|
| `NONE` | Valid settings message. The `settings` object describes the active stream structure. |
| `NOT_IN_SLM` | Basic Streaming is unavailable because the XL3 is not in Sound Level Meter mode. The settings message does not describe a valid active stream layout. |

`NOT_IN_SLM` is a stream state reported at the JSON level and is distinct from the startup text responses.

---

## Notes for Client Implementations

- The latest settings message is the source of truth for the active UI structure.
- The XL3 defines the stream structure; the client follows it.
- Do not assume fixed measurement labels, order, limits, timer context, or spectrum configuration (including `octres`).
- Tolerate `null` values and malformed numeric content without crashing.
- The website shall save the last used connection credentials, including connect key and password, and provide them as selectable entries in a quick connect list.
- Automatic reconnect behavior is out of scope for this document.

```
Socket opened.

```

```json

  "error": "NONE",
  "settings": {
```

### Error – wrong credential

```
Opening socket to "wss://connect.nti-audio.com/LX3VZ-V2FHU/api/live/"
Socket opened.
RX "Password:"
TX 1111
TX "START 123 Spectrum"
RX "Incorrect password"
Socket closed.
```

Result: startup failed. No application JSON stream is started.

Result: startup failed. No application JSON stream is started.
